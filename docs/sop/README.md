## How to Read an SOP

Each SOP follows this structure:
1. **Purpose** — Why this process exists
2. **Scope** — What triggers this process
3. **Process** — Step-by-step workflow
4. **Template** — Output format
5. **Ownership** — Who does what
6. **Cadence** — How often this runs
7. **Quality Standards** — How to validate output

---

## When to Create a New SOP

Create an SOP when:
- A process recurs across multiple phases or modules
- The process has a clear trigger and defined output
- Multiple workers need to follow the same process
- The process is not obvious from code alone

Do NOT create an SOP for:
- One-time tasks
- Bug fixes
- Feature development
- User documentation
